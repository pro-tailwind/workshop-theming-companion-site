You can see that we've followed Tailwind's convention of `*-50` to `*-900` for naming our shades.

We could use totally different shade names, but since we're going to implement multiple themes to _an already existing_ Tailwind CSS project, matching the naming convention will make our lives much easier.

## Current implementation

Right now, we're just defining three new colors in our Tailwind config:

```js
// tailwind.config.js
const themes = {...}

module.exports = {
  theme: {
    extend: {
      colors: {
        default: themes.default.colors,
        rainforest: themes.rainforest.colors,
        candy: themes.candy.colors,
      },
    },
  },
}
```

Hmmmm. Is this... theming?

<img src="/img/theming-butterfly.jpg" alt="butterfly meme about theming" />


Nope.


**There is no theming happeneing at all here**. We've extended our theme with new colors.

Each color is generating new, different utility classes. Like `bg-rainforest-500`, or `text-candy-800`.

Those have a "hardcoded" color baked in their name. They hold _one specific value_.

What we want is to be able to **use the same color utility class across all themes**. We need a name that _makes sesnse_ regardless of what color that utility class represents.

A _semantic_ color name.

And we need the colors to be _redefined_ based on the currently active theme.

We'll go for a CSS-only solution here, leveraging the great power of CSS custom properties, or CSS variables.

---

## CSS variables for the win

CSS variables are well supported in most browsers.

You don’t need anything special to start using them.

Matter of fact, Tailwind CSS makes extensive use of CSS variables already. If you’re using Tailwind, you are *already using* CSS variables.

What makes CSS variables super powerful is that they can be *redefined* within any CSS selector. When doing so, the new value will **only take effect within the scope of that selector**.

This is incredibly useful.

You can declare CSS variable at the top level (root scope) in your CSS. These values will be available everywhere.

You can then create “theme scope” selectors, where you redefine these same CSS variables with new values.

And any HTML element within a given theme scope will get those updated values when reading on of the CSS variables!
